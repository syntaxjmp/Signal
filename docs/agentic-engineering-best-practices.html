01

Foundational Principles

Agentic systems amplify both capability and risk. These principles form the bedrock of every design decision.

◯

Minimal Authority

Grant agents the least privilege required for each task. Scope tool access, API permissions, and file system reach per-invocation, not globally.

◆

Deterministic When Possible

Prefer deterministic logic for control flow and branching. Reserve LLM calls for tasks that genuinely require reasoning or generation.

△

Fail Loud, Recover Gracefully

Surface errors to the operator immediately. Design retry logic that degrades scope rather than retrying the same failing action endlessly.

■

Human-in-the-Loop by Default

Default to requiring approval for irreversible or high-blast-radius actions. Let users opt into more autonomy, never the reverse.

The Agent Contract

Every agent should have a clearly defined contract: what inputs it accepts, what tools it can use, what side effects it may produce, and what outputs it returns. This contract should be explicit in code, not implied by prompt text alone.

Principle

An agent that cannot explain what it is about to do and why should not be allowed to do it. Build introspection into the loop, not as an afterthought.

02

Agent Architecture Patterns

Choose the right architecture for the right level of complexity. Over-engineering agent systems is as dangerous as under-engineering them.

ReAct Loop (Reason + Act)

The foundational pattern. The agent reasons about the current state, selects an action (tool call), observes the result, and repeats until the task is complete or a termination condition is met.

  • When to use: Single-domain tasks with clear completion criteria (code generation, data retrieval, document analysis).
  • Strength: Simple to implement, easy to debug, predictable token usage.
  • Risk: Can loop indefinitely without proper exit conditions.
Pattern 
while not done:
    # 1. Reason about current state
    thought = llm.reason(context, observations)

    # 2. Select and execute action
    action = llm.select_tool(thought, available_tools)
    observation = execute(action)

    # 3. Update context and check termination
    context.append(thought, action, observation)
    done = check_completion(context, max_steps)

Plan-then-Execute

A planning phase generates a structured task list before execution begins. Each step is executed independently, with optional re-planning if a step fails.

  • When to use: Multi-step tasks where the full plan can be determined upfront (migrations, refactors, batch operations).
  • Strength: Predictable execution, easy to checkpoint and resume.
  • Risk: Plans may become stale if the environment changes during execution.

Router / Dispatch

A lightweight classifier routes incoming requests to specialized sub-agents. Each sub-agent owns a narrow domain and has access to only the tools it needs.

  • When to use: Systems handling diverse request types (support bots, developer tools, ops automation).
  • Strength: Separation of concerns, independent scaling and testing of each agent.
  • Risk: Misrouting can send tasks to the wrong agent; needs strong fallback handling.

Hierarchical (Orchestrator + Workers)

A coordinator agent decomposes complex goals into subtasks, delegates to worker agents, aggregates results, and handles cross-cutting concerns like error recovery.

  • When to use: Complex, multi-domain tasks where subtasks have dependencies (full-stack feature implementation, incident response).
  • Strength: Handles complexity that single agents cannot. Parallelism where subtasks are independent.
  • Risk: Coordination overhead, harder to debug, potential for cascading failures.
Rule of Thumb

Start with a single ReAct agent. Only add orchestration layers when you have evidence that a single agent cannot handle the task. Complexity should be earned, not assumed.

03

Tool Design & Integration

Tools are the agent's hands. Well-designed tools make agents reliable; poorly designed tools make them dangerous.

Design Principles

  • Single responsibility: Each tool does one thing. A search_files tool should not also modify files.
  • Descriptive schemas: Parameter names, descriptions, and examples in the tool schema are the primary way the agent understands when and how to use a tool. Invest heavily here.
  • Idempotent where possible: Calling the same tool with the same inputs should produce the same result. This makes retries safe and debugging easier.
  • Bounded output: Tools should return bounded, structured data. Never return unbounded text dumps — truncate, paginate, or summarize.
  • Error messages as guidance: When a tool call fails, the error message should tell the agent what went wrong and suggest corrective action.

Dangerous Tool Patterns

  • Shell escape hatches: A generic "run any bash command" tool gives the agent unlimited power. Scope it tightly or replace with purpose-built tools.
  • Mutable state without confirmation: Tools that write to databases, send messages, or deploy code should require explicit confirmation for destructive operations.
  • Unbounded recursion: A tool that can invoke the agent creates unbounded recursion risk. If self-invocation is needed, enforce depth limits.
Good Tool Schema 
{
  "name": "search_codebase",
  "description": "Search for code patterns across the repository. Returns
    matching file paths and line numbers. Use this before editing
    to locate the right files.",
  "parameters": {
    "pattern":    { "type": "string",  "description": "Regex pattern to match" },
    "file_glob":  { "type": "string",  "description": "Filter to specific files, e.g. '*.py'" },
    "max_results": { "type": "integer", "default": 20 }
  }
}
04

Memory & State Management

Agents without memory repeat mistakes. Agents with poorly managed memory hallucinate from stale context.

Memory Layers

  • Working memory (context window): The current conversation. Actively manage this — compress, summarize, or evict irrelevant information as the conversation grows.
  • Short-term memory (session): Task-scoped state like plans, intermediate results, and tool outputs. Store in structured formats (JSON, task lists) not prose.
  • Long-term memory (persistent): Cross-session knowledge like user preferences, project conventions, and learned corrections. Use file-based or vector stores with explicit read/write operations.
  • Semantic memory (retrieval): Embeddings over documentation, code, and past conversations. Use RAG patterns with careful relevance filtering.

State Management Rules

  1. Explicit over implicit: Never rely on the LLM "remembering" something from earlier in a long conversation. If state matters, write it down in a structured store.
  2. Read before write: Always check existing state before creating new entries. Duplicates and contradictions in memory degrade agent performance rapidly.
  3. Timestamp everything: Memory without temporal context is a liability. Stale memories are worse than no memories.
  4. Separate facts from interpretations: Store what happened and what was decided separately from why. The "why" changes; the "what" doesn't.
Signal Pattern

Signal's stateful memory service uses vector embeddings to find relevant context from past scans, then injects only high-similarity results into the agent's working context. This keeps the context window lean while preserving institutional knowledge.

05

Multi-Agent Orchestration

When a single agent isn't enough, orchestration becomes the critical engineering challenge.

Communication Patterns

  • Message passing: Agents communicate through structured messages with typed payloads. Each message has a sender, recipient, and schema-validated body.
  • Shared blackboard: Agents read and write to a shared state object. Simple but requires careful concurrency handling.
  • Event-driven: Agents subscribe to events and react independently. Excellent for loosely-coupled systems but harder to reason about ordering.

Coordination Rules

  1. Define handoff protocols: When Agent A delegates to Agent B, define exactly what context is passed, what Agent B is expected to return, and what happens if Agent B fails.
  2. Set budget limits: Every agent invocation should have a token budget, time budget, and step budget. Runaway agents are expensive and dangerous.
  3. Avoid shared mutable state: If two agents can write to the same resource, you will get conflicts. Use ownership semantics — only one agent writes to a given resource at a time.
  4. Parallelize independent work: If subtasks have no data dependencies, run them concurrently. But merge results carefully — conflicting outputs need resolution logic.
Orchestration 
// Hierarchical orchestration with budget enforcement
const orchestrator = new Agent({
  role: "coordinator",
  maxSteps: 25,
  maxTokens: 50_000,
  timeout: 120_000,
  workers: {
    researcher:  { tools: [search, read],  maxSteps: 10 },
    implementer: { tools: [read, edit],     maxSteps: 15 },
    reviewer:    { tools: [read, lint, test], maxSteps: 8 },
  },
  // Only the orchestrator can approve irreversible actions
  escalation: (action) => action.sideEffects.includes("write")
});
06

Safety, Guardrails & Alignment

Safety is not a feature — it's a property of the system. Every layer of the stack should contribute to it.

Defense in Depth

  1. Prompt-level guardrails: System prompts that define boundaries, refusal conditions, and escalation triggers. Necessary but not sufficient alone.
  2. Tool-level permissions: Each tool enforces its own access controls independent of the prompt. A "delete file" tool should verify the path is within the allowed directory regardless of what the agent says.
  3. Runtime monitors: Separate processes that observe agent behavior and can intervene — killing runaway loops, blocking suspicious tool calls, alerting operators.
  4. Output validation: Validate all agent outputs against schemas before they reach users or downstream systems. Reject malformed or out-of-scope responses.

Prompt Injection Defense

  • Treat all external data (user input, API responses, file contents, web pages) as untrusted.
  • Clearly delimit system instructions from user-provided content in the prompt.
  • Validate tool call arguments against expected schemas before execution.
  • Monitor for sudden behavior changes after processing external content.

Blast Radius Control

  • Sandbox execution: Run agent tool calls in isolated environments (containers, VMs, worktrees) whenever possible.
  • Reversibility bias: Prefer reversible actions. If an action is irreversible, require explicit human approval.
  • Incremental deployment: Roll out agent capabilities gradually. Start with read-only tools, then add write tools with approval gates, then reduce gates based on observed reliability.
Critical

Never trust an agent's self-assessment of safety. "I've verified this is safe" from an LLM is not verification. Use deterministic checks and human review for high-stakes actions.

07

Evaluation & Testing

Agentic systems are non-deterministic. Traditional unit tests are necessary but not sufficient.

Testing Layers

  • Unit tests for tools: Each tool should have deterministic tests. Mock the LLM, test the tool logic in isolation.
  • Integration tests for loops: Test the full agent loop with recorded LLM responses. Verify that the agent selects the right tools in the right order.
  • Eval suites for reasoning: Curated sets of tasks with known-good solutions. Run regularly, track pass rates and regressions across model versions and prompt changes.
  • Adversarial testing: Deliberately try to break the agent. Feed it conflicting instructions, malformed data, prompt injections, and impossible tasks.
  • Shadow mode: Run the agent in parallel with human operators. Compare decisions without executing the agent's actions.

Metrics That Matter

  • Task completion rate: What percentage of tasks are completed successfully without human intervention?
  • Steps to completion: Efficient agents complete tasks in fewer steps. Track median and p95.
  • Tool call accuracy: How often does the agent call the right tool with the right arguments on the first try?
  • Guardrail trigger rate: How often do safety guardrails fire? A rate near zero might mean they're not sensitive enough.
  • Cost per task: Total token usage and API costs per completed task. Optimize for both quality and efficiency.
08

Prompt Engineering for Agents

Agent prompts are different from chat prompts. They define behavior loops, not single responses.

System Prompt Structure

  1. Identity and role: Who is the agent? What domain does it operate in? What is its primary goal?
  2. Available tools: List tools with detailed descriptions, parameter docs, and usage examples. This is the most impactful section for tool selection accuracy.
  3. Operating rules: Constraints, permissions, and behavioral guidelines. Be specific: "do not modify files outside /src" is better than "be careful with files."
  4. Output format: Define exactly how the agent should structure its responses. Structured output (JSON, XML) reduces parsing errors.
  5. Examples: Show complete reasoning traces including thought, action, observation, and conclusion. Few-shot examples dramatically improve agent behavior.

Prompting Anti-Patterns

  • Vague instructions: "Be helpful" tells the agent nothing. "When the user asks to modify a file, first read the file to understand its structure, then make the minimal edit needed" tells it exactly what to do.
  • Contradictory rules: "Always ask for confirmation" + "minimize back-and-forth" creates confusion. Resolve contradictions explicitly by defining priority.
  • Over-reliance on prompt for safety: Prompts can be overridden. Use them for behavioral guidance, not as the sole safety mechanism.
09

Observability & Debugging

You cannot improve what you cannot see. Agentic systems require deeper observability than traditional software.

What to Log

  • Full reasoning traces: Every thought, action, and observation in the agent loop. This is your primary debugging tool.
  • Tool call metadata: Input arguments, execution duration, output size, success/failure, and any errors.
  • Token usage: Per-step and per-task token counts, broken down by input and output tokens.
  • Decision points: When the agent chose between multiple tools or approaches, log what it considered and why it chose what it did.
  • Guardrail activations: Every time a safety check fires, log the trigger condition and the action that was blocked.

Debugging Strategies

  • Replay from checkpoint: Serialize the agent's state at each step so you can replay from any point with modified prompts or tools.
  • Trace comparison: When a task fails, compare the trace against a known-good trace for a similar task. The divergence point reveals the issue.
  • Ablation testing: Remove individual tools or prompt sections to isolate which component is causing a behavior.
10

Production Operations

Running agents in production requires the same operational rigor as any critical system, plus additional considerations unique to non-deterministic software.

Operational Essentials

  • Rate limiting: Enforce rate limits on both inbound requests and outbound API calls. A single runaway agent can exhaust your LLM API budget in minutes.
  • Circuit breakers: If an agent or model provider starts failing, trip the circuit breaker and fall back to a degraded mode rather than retrying into a cascade.
  • Graceful degradation: Define what happens when the LLM is unavailable. Can the agent fall back to rule-based logic? Can it queue tasks for later?
  • Cost controls: Set per-user, per-task, and per-day spending limits. Alert when approaching limits. Kill tasks that exceed them.
  • Model version pinning: Pin the model version in production. Model updates can change agent behavior in unexpected ways. Test new versions in staging first.

Deployment Strategy

  1. Canary deployments: Route a small percentage of traffic to new agent versions. Compare performance metrics before full rollout.
  2. Feature flags per tool: Enable new tools gradually. Start disabled, enable for internal users, then expand.
  3. Rollback plan: Every deployment should have a one-command rollback. Agent behavior changes can be subtle — fast rollback is critical.
11

Anti-Patterns to Avoid

Learn from the mistakes that have cost teams the most time, money, and reliability.

Anti-Pattern Better Approach
Giving the agent unlimited shell access Purpose-built tools with scoped permissions per operation
One massive system prompt with every rule Modular prompt composition — load relevant sections based on task type
Relying on the LLM to "remember" state Explicit state management with structured stores and read-before-act patterns
Retrying failed actions in a loop Diagnose root cause, try alternative approach, or escalate to human
Building a multi-agent system from day one Start with one agent, add orchestration only when single-agent approach provably fails
Testing only happy paths Adversarial evals, edge cases, prompt injection attempts, and impossible tasks
Trusting the agent's self-reported confidence Deterministic validation, schema checks, and human review for high-stakes outputs
No spending limits on LLM API calls Per-task and per-day budgets with hard kill switches
Logging only inputs and outputs Full reasoning traces, tool call metadata, and decision-point annotations
Deploying new model versions without staging Pin model versions, test in staging, canary deploy, then promote
12

Ship-Ready Checklist

Before deploying an agentic system to production, verify every item.

Architecture

  • Agent contract is explicitly defined (inputs, tools, side effects, outputs)
  • Architecture pattern matches task complexity (not over-engineered)
  • Termination conditions exist for every loop (step limit, token limit, timeout)

Tools

  • Each tool has a single responsibility and a descriptive schema
  • Destructive tools require confirmation gates
  • Tool outputs are bounded and structured
  • Error messages guide the agent toward corrective action

Safety

  • Minimal-authority principle applied to all tool permissions
  • External data treated as untrusted (prompt injection defense)
  • Irreversible actions require human approval by default
  • Runtime monitors can kill runaway agents
  • Output validated against schemas before delivery

Observability

  • Full reasoning traces logged for every agent invocation
  • Token usage and cost tracked per task
  • Guardrail activations logged and alerted on
  • Replay-from-checkpoint capability for debugging

Operations

  • Rate limits on inbound and outbound API calls
  • Per-task and per-day cost limits with hard kills
  • Model version pinned and tested before promotion
  • One-command rollback available
  • Graceful degradation defined for LLM outages

Testing

  • Unit tests for every tool
  • Integration tests for the full agent loop
  • Eval suite with pass rate tracking across versions
  • Adversarial testing performed and documented
